01 · The Problem

My planning life was scattered across 5 apps

As someone managing a job search, a Master's thesis, fitness goals, and a side project simultaneously, I found myself constantly context-switching — checking Notion for goals, Google Calendar for schedule, browser bookmarks for events, and a notes app for reflections. Nothing talked to each other.

  • 📭
    No unified command center

    Daily tasks, yearly goals, and discovery lived in separate apps with no connection between them.

  • 🔀
    Context-switching overhead

    Deciding whether to attend an event required opening 3 apps to check schedule, goals, and energy levels.

  • 🧠
    No visibility on learning progress

    Goals like "complete IBM certificate" or "finish thesis" had no visual progress tracking tied to daily actions.

Core Job-to-be-Done

"When I wake up or check my phone between tasks, I want to immediately know what I should do next — without mentally stitching together information from five different places."

02 · Product Thinking

User stories that drove prioritization

I wrote user stories from my own perspective first, then categorized them using MoSCoW to define the MVP scope.

US-01 As a user, I want to see all my tasks for today in one view so I can start my day with clarity. Must
US-02 As a user, I want to paste an event URL and get a decision card (add/ignore) so I can process new activities without leaving the app. Must
US-03 As a user, I want the app to learn my preferences so it can recommend activities similar to ones I've accepted before. Should
US-04 As a user, I want to see my learning goals as a visual map so I can feel the structure and progress of what I'm studying. Could
US-05 As a user, I want to break yearly goals into milestones and see percentage completion so I don't lose sight of the big picture. Should

03 · Execution

Shipped in two phases

I intentionally scoped Phase 1 to the two highest-friction problems — daily planning and event discovery — before building the more exploratory Phase 2 features.

Phase 1 · MVP
Daily Calendar + Web Scraper
Built the month-view calendar, task CRUD, and the full scraper decision flow. Established localStorage persistence and Zustand state management.
Phase 1 · Extension
Smart Suggestion Engine
Added behavioral data tracking — accept/ignore actions are stored per category and used to surface recommendation reasons on future scrapes.
Phase 2
Yearly Goals + Visual Mind Map
Added the goal tracker with milestone checklists and built the interactive SVG mind map from scratch — no graph library used.
Phase 3 · Planned
Cloud Sync + AI Integration
Cross-device sync via Supabase, AI-powered daily plan generation, and real web scraping replacing the current mock.

Key Decisions

Decision 01
Database: Local vs. Cloud
localStorage VS Supabase

Chose localStorage to ship faster with zero auth complexity. The real product need (persistence) is met; cloud sync is a Phase 3 enhancement.

Decision 02
Mind Map: Library vs. Custom SVG
Custom SVG VS D3 / Cytoscape

Custom SVG gave full control over the progress ring animation and interaction model. The constraint forced a deeper understanding of what the feature actually needs.

Decision 03
Scraper: Mock vs. Real
Mock scraper VS Real HTTP fetch

A real scraper requires a backend proxy to bypass CORS. The mock validates the entire UX flow and decision logic — the right thing to prove in a prototype.

04 · Features

What was built, and why

📅
Daily Calendar + Timeline View

Month-view calendar with color-coded task dots. Day view toggles between a task list and an hour-by-hour timeline (6:00–24:00) showing scheduled blocks and free slots at a glance. Full task CRUD with inline edit, time, duration, priority, and category fields.

PM Rationale

The timeline view answers the question "when am I free?" directly — the absence of blocks is as informative as the blocks themselves. Toggling between list and timeline serves two mental modes: planning and scheduling.

🌐
Web Scraper & Action Center

Paste any event URL — the app simulates extracting activities and presents them as decision cards with an Accept or Ignore binary choice. Accepted items go directly into the daily plan.

PM Rationale

The key insight: the bottleneck isn't finding events, it's deciding quickly. The two-button decision flow reduces cognitive load to a binary choice, removing the "I'll do it later" trap.

✨
Smart Suggestion Engine

Behavioral learning loop — every accept/ignore action is stored per category. On future scrapes, the system surfaces a "recommendation reason" if the activity matches a user's historical preferences.

PM Rationale

This is the data flywheel: more usage → better signals → better recommendations → more value → more usage. Even with simple logic, it demonstrates the personalization principle central to modern productivity products.

🗺️
Visual Mind Map

Interactive SVG node map with animated progress rings (stroke-dasharray technique), zoom/pan canvas, radial layout algorithm, and click-to-expand subtask detail panel — zero external graph libraries.

PM Rationale

Linear lists make learning goals feel like tasks. The spatial, radial map activates a different mental model — you see relationships and structure, not just checkboxes. Progress rings give emotional feedback at a glance.

🎯
Yearly Goal Tracker

Goal cards with category, target date, description, and milestone checklists. Completing milestones auto-calculates overall goal progress percentage. Full inline edit and delete on every goal and milestone.

PM Rationale

Most goal apps fail because goals aren't connected to daily behavior. This tracker sits in the same app as the daily calendar — creating a natural loop between "what do I do today" and "why am I doing it."

🔒
Privacy-First Share View

A dedicated /share route renders the same calendar data with all task details stripped — every block appears as "Booked" with only the time slot visible. Designed for sharing availability without exposing personal details.

PM Rationale

Sharing a calendar is a social act; sharing what's inside is a privacy risk. Separating the two mirrors how enterprise calendar tools handle free/busy access control, and sets up a clean boundary for Phase 3 multi-user sync.

🗑️
Trash & Restore

Deleting a task or goal moves it to a dedicated Trash page instead of permanent removal. Items can be restored to their original state or permanently deleted. Sidebar badge shows the current trash count.

PM Rationale

Destructive actions without undo are a trust liability — one accidental delete erodes confidence in the whole system. The two-step trash pattern (soft delete → explicit purge) is the industry standard for a reason: it makes users feel safe to act quickly.

05 · If This Were a Real Product

What I'd measure

As a PM, shipping isn't the end — it's the beginning. Here are the metrics I'd track post-launch to validate whether Pulse Plan is actually reducing planning friction.

📊
Daily Active Rate

Are users returning every day? Planning tools only have value if they're habitual. Target: 60% D7 retention.

✅
Task Completion Rate

Percentage of daily tasks marked done. The core proxy for whether the app is helping users follow through.

⚡
Scraper Accept Rate

What % of scraped activities get added to the plan? Rising rate over time validates the suggestion engine is learning.

🗺️
Mind Map Node Depth

Average subtasks per node. Depth signals that users are investing in the tool for real learning goals, not just exploration.

🎯
Milestone Velocity

Rate at which users complete milestones over time. A flattening curve would signal that goal-setting is too ambitious or unclear.

🔗
Cross-feature Usage

% of users who use 3+ features. The app's value compounds when features interact — this measures whether the "unified" premise actually holds.

07 · Reflection

What I learned, and what comes next

✓ What I Learned
  • Scoping the MVP strictly to must-haves made Phase 1 shippable in days instead of weeks.
  • Writing user stories first — even for a solo project — forced clarity on what "done" looks like.
  • A behavioral data loop doesn't need ML to be valuable; even simple counters create meaningful personalization signals.
  • Building the mind map from scratch was harder but taught me that constraints drive creative solutions.
  • HCI background made me obsess over interaction feedback — every button needed a state transition, every action needed a visual response.
  • A timeline view and a list view serve different mental modes — giving users both costs one toggle and removes a whole category of friction.
  • Soft-delete (trash) is a trust feature, not just a safety net. Users act more confidently when they know actions are reversible.
  • Designing a privacy boundary early (share view showing only "Booked") made the architecture cleaner for future multi-user sync.
→ What I'd Do Next
  • Interview 5 other students about their planning pain points before building Phase 3 — I may be solving for myself, not a broader segment.
  • Replace the mock scraper with a real backend proxy and test against actual event sites (Eventbrite, Meetup, lu.ma).
  • Add Supabase for cloud sync — the #1 blocker for real daily use is being tied to one device. The share view is already architected for this.
  • A/B test the scraper decision card layout — does showing the recommendation reason before or after the title affect accept rate?
  • Build a weekly review flow: what did you plan vs. what did you do? Closing the feedback loop is where planning tools earn long-term trust.
  • Auto-expire trash after 30 days — mirrors the behavior users expect from tools like Notion and Gmail, reducing mental overhead.
